# homebrain 项目框架

> 目录结构、各模块职责、数据流、依赖方向。配套见 [实施计划](./implementation-plan.md)。

## 整体结构

`homeagent`（家庭编排层）→ `homebrain`（gbrain 薄封装）→ `gbrain`（CLI 子进程）。单向依赖：上层只认下层接口，gbrain 升级/换 MCP 后端时只改 homebrain 一层。

## 模块职责

### packages/homebrain — gbrain 薄封装
全项目**唯一**接触 gbrain 的地方。集成边界 = gbrain CLI 子进程（gbrain 库层没有 capture/query/dream 这些"友好动词"，只在 CLI/MCP 层）。

`createHomebrain(config)` 暴露：

| 方法 | 作用 | 底层 |
|---|---|---|
| `remember` | 写一条已抽取的事实到某成员名下 | `gbrain capture --source ... --json`（写串行化；member/tags/occurredAt 作为 capture 文本 front matter 保留） |
| `ask` | 综合问答（带引用） | `gbrain query` 文本输出（当前 contract 从 search 风格行抽 citations） |
| `search` | 原始检索（无 LLM） | `gbrain search` 文本输出 |
| `recall` | 按时间检索（那年今日/周报） | 当前为 `search + YYYY-MM-DD 解析过滤` fallback；gbrain 原生时间检索待实测 |
| `upsertProfile`/`getProfile` | 成员画像读写 | `gbrain put/get partners/<slug>/user`（对应 `partners/<slug>/USER.md`） |
| `runDreamCycle` | 后台 sync + consolidate/synthesize | 可先 `sources list/add` 确保本地 source path；再 `gbrain sync` + `gbrain dream --json`；按 JSON report status 归一成败 |
| `health` | 健康检查 | `gbrain --version` |

**写串行化**（`serialize.ts`）：gbrain 底层 PGLite 是单写者，所有写操作排进同一 FIFO 队列，否则并发写争用写锁报错。读操作不受限。

**gbrain contract**（`gbrainCli.ts`）：集中处理 gbrain CLI 的命令构造、JSON / 文本输出归一、`sources list/add`、stdout/stderr 错误收敛。Slice 0 实测出的输出形状必须先落成 fixture / contract 测试，再让 `index.ts` 的家庭记忆 interface 消费，避免 gbrain breaking change 泄漏到 homeagent。当前实测记录见 [Slice 0 gbrain contract](./slice0-gbrain-contract.md)。`GBRAIN_SOURCE_PATH` 存在时，homebrain 会在 sync/写入前确保本地 source 已注册；若 `default` 已存在但没有 `local_path`，不会自动删除重建，而是要求改用新 source 或手工处理。

只认 homebrain 接口，永不直接 spawn gbrain。四块 + 任务 + 运行时：

- **① connectors**：`Connector` 接口 = `receiveMessages(): AsyncIterable<IncomingMessage>` + `sendMessage()`。平台细节藏在 adapter（飞书/CLI/未来微信）。`cli.ts` 已实现（本地调试，不依赖飞书）；`feishu.ts` 已封装 `lark-cli event consume im.message.receive_v1 --as bot` 入站 NDJSON（兼容扁平与原始嵌套事件结构，text/post 事件会提取正文并识别 @bot，图片/文件事件以及富文本 post 内嵌 `img/media` 会保留 attachment key/url/name/localPath/extractedText 而不是把 content JSON 当文本；配置 `FEISHU_ATTACHMENT_DOWNLOAD_DIR` 后会通过 `lark-cli im +messages-resources-download` 下载附件到本地相对路径）、异常退出重启与 `lark-cli im +messages-send --chat-id ... --text ... --as bot` 出站命令；Feishu connector 启动前会校验 EventKey 与 bot open_id，后者用于 @bot 识别与标签剥离；真实权限/EventKey 闭环与附件真实下载权限待实测。
- **② understanding**：`MemoryExtractor` seam（passthrough + Claude JSON 抽取基础版，LLM 调用前会跳过纯短寒暄；附件元数据 key/url/name/localPath/extractedText 会进入抽取 prompt，附件 fact 会自动补 `attachment/image/file` 标签；本地文本附件抽取 seam 已接入，入口默认读取安全相对路径中的 txt/md/csv/json/log 前缀；显式开启 `HOMEAGENT_IMAGE_OCR_ENABLED` 时会用 Claude Vision 对本地图片做文字提取，OCR 返回“无”时不写入附件正文；Claude client 经统一 factory 包上本地 JSONL 日志与每日预算估算/阻断，并复用于 extractor/planner/图片 OCR；真实样本效果待 provider key 验证）+ router（已判定 被动记录 / @提问 / 任务目标 / 任务反馈 / 任务暂停 / 任务恢复）。
